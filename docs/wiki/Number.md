# number 타입

`number`는 Bpp에서 컴파일러가 특별히 아는 기본 숫자 타입입니다. 소스
코드에서는 내장 타입처럼 쓰지만, 실제로는 표준 라이브러리의 `Number`
타입으로 해석됩니다.

목표는 일반 사용자 코드에서 정수, 실수, 복소수 연산을 할 때 `i64`,
`u64`, `f64`, 큰 정수 타입을 직접 고르지 않아도 되게 만드는 것입니다.
ABI, 비트 레이아웃, 극한 성능 루프가 중요한 저수준 라이브러리 코드는
여전히 원시 숫자 타입을 직접 사용할 수 있습니다.

## 지원 값

현재 `number`는 다음 값 계층을 지원합니다.

- 정수 값:
  - 작은 부호 있는 즉시값 정수
  - `i8/i16/i32/i64`, `u8/u16/u32/u64` 생성자 경로
  - 작은 정수 범위를 넘으면 자동으로 승격되는 크기 제한 없는 큰 정수 저장값
- 실수 값:
  - `Number.from_f64`와 문맥 기반 `number` 변환을 통한 `f64` 저장값
- 복소수 값:
  - `Number.from_complex(real, imag)`
  - `7i` 같은 허수 접미사 리터럴
  - `76 + 7i` 같은 표현식
- 정확 확장 값:
  - `Number.from_rational(num, den)`
  - `Number.from_decimal(coeff, scale)`
  - `Number.from_bigfloat(mant, exp, precision)`

예시:

```bpp
func main() -> u64 {
    var a: number = 123;
    var b: number = 1.5;
    var z: number = 76 + 7i;

    var half: number = Number.from_rational(1, 2);
    var cents: number = Number.from_decimal(125, 2); // 1.25
    var tiny: number = Number.from_bigfloat(1, -3, 128); // 1 * 2^-3

    var s: number = half + cents + tiny;
    return 0;
}
```

## 산술 의미

기본 산술은 숫자 계층 전체에서 동작합니다.

- `+`, `-`, `*`, `/`는 정수, 실수, 복소수, 정확 확장 값에서 동작합니다.
- 양쪽 피연산자가 정수이면 `/`는 기존 정수 몫 의미를 유지합니다.
- 유리수/십진수/큰 실수끼리의 산술은 가능한 한 정확한 분수 형태로
  정규화합니다.
- `f64` 실수 값과 섞이면 근사 실수 경로를 사용합니다.
- 복소수 나눗셈은 일반적인 복소수 나눗셈 공식으로 처리합니다.

유리수, 십진수, 큰 실수 값은 다음처럼 정규화됩니다.

- 유리수의 분모는 양수로 유지됩니다.
- 유리수 값은 가능한 경우 약분됩니다.
- 십진수 값은 뒤쪽 10진 스케일의 0을 가능한 만큼 줄입니다.
- 큰 실수 값은 가수에서 2의 거듭제곱 인자를 가능한 만큼 분리합니다.

## 비교

동등 비교와 순서 비교는 의도적으로 분리되어 있습니다.

- `==`, `!=`는 복소수를 포함한 모든 `number` 값에서 동작합니다.
- `<`, `<=`, `>`, `>=`는 순서가 정의되는 실수 계열 숫자 값에서 동작합니다.
- 허수부가 0이 아닌 복소수 값에 순서 비교를 시도하면 런타임에서 panic합니다.

예시:

```bpp
var z: number = 1 + 2i;
var w: number = Number.from_complex(1, 2);

if (z == w) {
    // 가능
}

// z < w 는 panic합니다. 복소수에는 표준 전체 순서가 없기 때문입니다.
```

## 정수 전용 연산자

다음 연산자는 정수 `number` 피연산자만 허용합니다.

- `%`
- `&`, `|`, `^`, `~`
- `<<`, `>>`

실수, 복소수, 유리수, 십진수, 정수가 아닌 큰 실수 값에 사용하면 panic합니다.
비트 레이아웃이나 기계어 단위 자체가 중요한 코드는 원시 정수 타입을 직접
사용하는 편이 맞습니다.

## 변환

검사 변환 함수는 원시 타입 경계를 명시적으로 만듭니다.

- `to_i64_checked_value()`
- `to_u64_checked_value()`
- `to_i8_checked_value()`, `to_i16_checked_value()`, `to_i32_checked_value()`
- `to_u8_checked_value()`, `to_u16_checked_value()`, `to_u32_checked_value()`

정확 확장 값은 실제 값이 정수일 때만 정수 타입으로 변환됩니다.
근사 변환이 괜찮은 경우에는 `to_f64_lossy()` 같은 근사 변환 함수를 사용할 수
있습니다.

`to_str()`는 정수, 실수, 복소수, 유리수, 십진수, 큰 실수 저장값을 지원합니다.

## 최적화

`number`는 컴파일러 계층과 런타임 계층 양쪽에서 최적화되어 있습니다.

컴파일러 쪽 최적화:

- `number`는 컴파일러가 특별히 아는 기본 타입입니다. 파서, 타입 정보,
  중간 변환, 기존 코드 생성기, SSA 코드 생성기, LLVM 빌드 경로가 `number`
  포장과 런타임 보조 함수 호출을 알고 있습니다.
- `i8/i16/i32/u8/u16/u32` 작은 정수 생성자는 가능한 경우 인라인 태그 값으로
  방출됩니다.
- 정수 리터럴과 생성자 호출은 `number`가 되기 전까지 더 좁은 부호 있음/없음
  의도를 보존할 수 있습니다.
- 컴파일러가 표현 가능성을 증명한 경우, `number` 저장소는 컴파일러가 알고
  있는 좁은 정수/f64 경우로 특수화될 수 있습니다. 그렇지 않은 일반 스택 값
  표현은 8바이트 태그 `Number`입니다.
- 작은 정수 산술과 비교는 생성 코드에 빠른 인라인 경로가 있습니다.
  값이 작은 즉시값이 아니거나 연산이 작은 정수 범위를 넘을 때만 런타임
  보조 함수로 넘어갑니다.

런타임 쪽 최적화:

- 작은 부호 있는 정수는 힙 할당 없이 태그가 붙은 즉시값으로 저장됩니다.
- 더 큰 정수는 자동으로 큰 정수 저장값으로 승격됩니다.
- 큰 정수의 자리 단위는 기수 `2^31`을 사용합니다. 그래서 자리 곱셈과 올림
  처리가 `u64` 안에 머뭅니다.
- 큰 정수 곱셈은 여러 알고리즘을 사용합니다.
  - 작은 입력: 기본 자리별 곱셈
  - 중간 크기의 균형 잡힌 입력: Karatsuba
  - 더 큰 균형 잡힌 입력: Toom-3
  - 충분히 큰 입력 중 내부 NTT 길이 한도에 들어오는 경우: NTT 기반 곱셈
  - 내부 NTT 길이 한도를 넘는 경우: 메모리 사용과 변환 크기를 제한하기 위해 Toom-3 경로로 되돌아감
  - 이 계층은 일반적인 큰 정수 연산을 빠르게 처리하기 위한 현재 구현 전략이며, 무한히 큰 입력까지 항상 최선이라는 뜻은 아닙니다.
- 큰 정수 나눗셈은 최적화된 경로를 사용합니다.
  - 작은 나누는 수 전용 나눗셈
  - Knuth 방식의 긴 나눗셈
  - 큰 입력용 분할 정복 나눗셈
- 정확 확장 숫자 값은 불필요한 분모, 십진 스케일, 가수 인자를 들고 다니지
  않도록 정규화합니다.

## 실전 가이드

일반 숫자 코드에서는 `number`를 기본값으로 사용해도 됩니다.

- `i64`를 넘을 수 있는 계수기
- 사용자 입력 기반 산술
- 정수/실수 혼합 계산
- 복소수 계산
- 기계어 단위 속도보다 정확성이 중요한 유리수/십진수 스타일 계산

원시 숫자 타입을 직접 쓰는 편이 나은 경우도 있습니다.

- ABI와 직접 맞닿는 코드
- 포인터 크기 값과 메모리 레이아웃
- 비트 마스크와 하드웨어에 가까운 연산
- 고정 원시 타입 자체가 성능 계약인 극한 반복문
- 표준 라이브러리/런타임 내부 구현

요약하면 `number`는 고수준 기본 숫자 타입입니다. 프로그램이 명시적으로
기계 표현을 다루는 경우에는 원시 정수와 부동소수점 타입이 여전히 더 정확한
도구입니다.
