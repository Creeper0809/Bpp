# Lexical Rules

이 페이지는 토큰화(lexer) 기준의 문자 규칙을 정리합니다.

## Why It Exists

- 파서는 토큰을 입력으로 동작합니다.
- 따라서 문법 문제를 디버깅할 때 먼저 "문자가 어떤 토큰으로 인식되는지"를 알아야 합니다.

## How to Use

### Comments

- 한 줄 주석: `// ...`
- 블록 주석: `/* ... */`

```bpp
// single line
/* block comment */
```

### Identifiers

- 일반적으로 영문/숫자/밑줄 패턴을 사용합니다.
- 키워드와 충돌하는 이름은 식별자로 사용할 수 없습니다.

### Literals

- 정수: `123`
- 실수: `3.14`
- 문자: `'a'`, `'\n'`
- 문자열: `"text"`
- 불리언: `true`, `false`
- null: `null`

### Delimiters and Operators

- 구분자: `() { } [ ] , ; :`
- 산술: `+ - * / %`
- 비교: `== != < <= > >=`
- 논리: `&& || !`
- 비트: `& | ^ ~ << >>`
- 대입: `= += -= *= /= %=`
- 기타: `.`, `.?`, `@`, `->`, `?`

## Constraints (v11)

- 블록 주석은 중첩 보장을 전제로 하지 않습니다.
- 잘못된 문자/리터럴 포맷은 lexer 단계에서 즉시 실패합니다.
- 일부 토큰(`match`)은 parser에서 기존 노드로 재해석(alias)될 수 있습니다.

## Cautions

- char literal은 1문자 규칙/escape 규칙을 지켜야 합니다.
- 문자열/주석 미종료는 parser가 아니라 lexer 오류입니다.
- "파서 에러처럼 보이는 문제"가 실제로는 토큰화 실패인 경우가 많습니다.

## Practical Tips

- 실패 위치의 line/column을 먼저 확인합니다.
- 의심 구간 앞뒤로 토큰 경계를 만들기 위해 괄호/세미콜론을 임시 추가해 재시도하면 원인 파악이 빠릅니다.
